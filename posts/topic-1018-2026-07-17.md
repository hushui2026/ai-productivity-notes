# 3天写崩十几个功能？非程序员用Cursor独立开发的正确工作流

上周我把自己做的一个小应用发到部门群里，好几个同事私信问我：你不是做运营的吗，什么时候学会写代码了？

我说，我没学。

然后就是一连串的追问。干脆写出来，有些坑你们能少走。

---

**你到底用什么做的？**

Cursor。简单说它是个代码编辑器，里面内置了 AI Agent。你可以理解为 ChatGPT 跟你的项目文件打通了，它能直接读你的代码、改你的代码、还能帮你跑起来。

ChatGPT 也能写代码，但你得把代码复制来复制去，改一处报一处，来回切窗口，脑子会炸。Cursor 不用，它直接在你项目里干活。

**完全不懂代码也能用？**

说实话，完全不懂的话，第一步就卡住。

你不需要会写，但得能看懂大概。比如知道什么是前端什么是后端，知道 HTML 是管页面结构的，CSS 是管样式的，JavaScript 是管交互的。这些东西让 AI 给你解释一遍，花十分钟就能建立基本概念。

我是产品运营出身，之前连 Python 和 Java 都分不清。但用了两周之后，我现在能大致看懂代码逻辑，知道哪里是数据请求，哪里是页面渲染。够用了。

**具体怎么开始？工作流是什么？**

这个是重点。我踩了最大的坑就是一开始直接跟 Cursor 说"帮我做一个 XX 应用"，然后它给我生成了一坨代码，跑不起来，我也不知道哪错了。

正确的做法是拆。把一个应用拆成很多个小任务，一个一个喂给 AI。

我的流程大概是这样：

先想清楚要做什么。不是脑子里想，是写下来。写一个文档，把功能列出来，每个功能用户怎么操作、看到什么、点了之后发生什么。

然后让 Cursor 帮你拆技术任务。这一步很关键，因为你不知道一个功能在技术上要分成几步实现。

Prompt 大概是这样：

```
我要做一个 [你的应用名称]，这是一个 [一句话描述应用用途] 的应用。

核心功能：
1. [功能1：用户做什么，看到什么]
2. [功能2：用户做什么，看到什么]
3. [功能3：用户做什么，看到什么]

我是非技术背景，请帮我把这个项目拆解成可独立开发和测试的技术任务，每个任务要足够小，小到可以一次完成并验证。按开发顺序排列，告诉我每个任务完成后应该能看到什么效果。
```

Cursor 会给你一个任务清单。然后你就一个一个来。

**一个一个来是什么意思？**

就是第一轮对话让它做任务一，做完了你看看效果对不对。对了，再开一个新对话让它做任务二。不对，就在当前对话里让它改。

为什么要开新对话？因为对话太长了之后 AI 会犯糊涂，前面说的东西它会忘或者混在一起。一个任务一个对话，干净。

这中间有个关键操作：每次 AI 生成代码后，你让它跑一下看看报不报错。Cursor 里有终端，可以直接运行。报错了就把错误信息丢给它，让它自己修。

**报错了你也看不懂错误信息怎么办？**

直接复制错误信息粘贴回去。Cursor 的 Agent 模式能自己读终端输出，你甚至不用说"帮我看看这个报错"，它自己就会去修。

这个体验挺魔幻的。你坐在那儿，看 AI 自己报错、自己改、再跑、再报错、再改。有时候能来回折腾十几轮。但它最终能给你跑通。

不过有个前提：你的任务得拆得够小。任务太大，报错链就长，它改着改着可能跑偏了，把前面好的地方也改坏了。我之前就遇到过一个登录功能，AI 改了半天，最后把首页样式也搞乱了。后来说不通了，回退重来。

**有没有什么特别需要注意的？**

有几个。

第一，项目结构要让 AI 先建好。第一轮对话别急着写功能，先让它搭项目骨架。目录结构、依赖安装、基本配置，这些东西一次性搞对，后面会顺很多。

```
我要用 [技术栈，比如 React + Node.js] 开发一个 [应用名称]。
请先帮我搭建项目基础结构，包括：
- 合理的目录结构
- 必要的依赖和配置文件
- 一个能跑起来的空项目（比如一个空白页面 + 本地服务）
告诉我每一步怎么运行，验证项目能正常启动后再继续。
```

第二，版本管理一定要开。就是 git。这个东西你现在不会没关系，让 Cursor 教你。核心操作就三个：提交、回退、看历史。每完成一个小任务就提交一次，出问题就回退。这是你的保险绳。

别问我怎么知道不提交会出事。某天 Cursor 改崩了我连回退的地方都没有，坐在工位上一言不发，那种感觉你体会一次就记住了。

第三，别贪多。你看到 AI 一下能生成那么多代码，很容易飘，觉得啥都能做。但功能堆多了之后，互相之间会打架。先把核心功能跑通，再加锦上添花的东西。

我第一个应用就是栽在这，三天加了十几个功能，最后整个应用跑不起来了，还查不出哪里的问题。推倒重来。

**你说重构经典应用，具体做了什么？**

我重新做了个待办清单应用。市面上这种工具很多，但每个都有我不满意的地方。我自己要的那种很简单：能按场景分（工作、生活、学习），能设优先级，能拖拽排序，最好还能跨设备同步。

这么个小东西，我断断续续花了两周。其中第一周基本在踩坑和返工，第二周才真正顺了。如果让我从头再来一次，可能三四天就能搞定。

**最后一个问题，这种开发方式的天花板在哪？**

说实话我还没摸到。但能感觉到的是，越复杂的交互逻辑，AI 越容易翻车。比如涉及实时通信、复杂动画、多用户权限管理这些，它容易顾此失彼。

但常规的增删改查、表单、列表、页面跳转这些，基本能一把过。做个内部工具、管理后台、小程序，这种需求 AI 完全能扛。

我现在觉得真正的瓶颈不是技术，是你有没有想清楚要做什么。AI 写代码很快，但你需求不清晰的时候，它会给你造一堆你不需要的东西，然后你还得删掉重来。

想清楚比写出来难。这个道理跟用不用 AI 没关系。
